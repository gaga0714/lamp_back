# 管理员端功能精简方案

## 1. 目标

管理员账号登录后，仅保留管理员职责相关功能，去掉不属于管理员角色的学生端个人功能，并消除重复的实验室管理入口。

本次调整目标如下：

- 管理员不需要：
  - 考勤签到
  - 考勤记录
  - 请假申请
  - 我的请假
  - 我的预约
- 管理员保留：
  - 考勤管理
  - 实验室列表
  - 预约审批
  - 实验室管理
  - 用户管理
  - 个人资料 / 修改密码
- 管理员端“实验室管理”只保留一个入口

## 2. 当前现状

### 2.1 已经符合预期的部分

根据当前前端路由配置，以下页面已经通过 `meta.roles` 限制为仅 `student` 可见，因此管理员默认不会在左侧菜单中看到：

- `考勤签到`
- `考勤记录`
- `请假申请`
- `我的请假`

对应文件：

- `D:\bishe\lamp_front\src\router\routes.js`

当前配置已经是：

- `/attendance/check-in` -> `roles: ['student']`
- `/attendance/records` -> `roles: ['student']`
- `/attendance/leave/apply` -> `roles: ['student']`
- `/attendance/leave/list` -> `roles: ['student']`

同时，后端考勤请假接口也已经做了角色限制，因此管理员即使手动访问这些接口，也会被拒绝。

### 2.2 仍然存在的问题

#### 问题 1：管理员仍然能看到“我的预约”

当前路由中：

- `D:\bishe\lamp_front\src\router\routes.js`

`/lab/my-bookings` 没有设置 `meta.roles`，因此所有登录用户都能在菜单中看到“我的预约”，管理员也会显示。

同时工作台中：

- `D:\bishe\lamp_front\src\views\dashboard\DashboardView.vue`

无论当前角色是什么，底部都统一展示：

- `实验室列表`
- `我的预约`

这会导致管理员在工作台中仍然出现“我的预约”按钮。

后端方面：

- `D:\bishe\lamp_back\src\main\java\com\lamp\controller\LabController.java`

当前以下接口仅做了“登录校验”，没有做角色限制：

- `POST /api/lab/booking`
- `GET /api/lab/booking/my`
- `PUT /api/lab/booking/{id}/cancel`

因此管理员不仅会看到“我的预约”，理论上还可以实际创建预约、查看自己的预约和取消预约，这与“管理员不需要我的预约”的目标不一致。

#### 问题 2：实验室管理入口重复

当前前端有两个管理员实验室管理入口：

1. `/lab/manage`
   - 组件：`D:\bishe\lamp_front\src\views\lab\LabManageView.vue`
   - 标题：`实验室管理`

2. `/admin/lab`
   - 组件：`D:\bishe\lamp_front\src\views\admin\LabManageView.vue`
   - 标题：`实验室管理`

其中 `src/views/admin/LabManageView.vue` 只是一个跳转壳页面，内容是提示“前往实验室管理”，本身并不承载真正的管理功能，因此会在管理员菜单中形成两个“实验室管理”，造成重复和混淆。

## 3. 调整原则

本次方案遵循以下原则：

1. 前端菜单隐藏和前端路由限制要同步进行
2. 后端接口权限也要同步收口，避免“菜单不显示但接口还能调用”
3. 对于重复功能，只保留真正承载业务逻辑的页面，删除壳入口
4. 尽量少改业务逻辑，只做角色边界和入口收敛

## 4. 完整实现方案

### 4.1 去掉管理员的“我的预约”

#### 前端改造

修改文件：

- `D:\bishe\lamp_front\src\router\routes.js`
- `D:\bishe\lamp_front\src\views\dashboard\DashboardView.vue`
- `D:\bishe\lamp_front\src\router\index.js`（如已有角色守卫则无需额外逻辑，只需路由配置收紧）

调整内容：

1. 给 `/lab/my-bookings` 增加角色限制：
   - 建议改为 `roles: ['student', 'teacher']`
   - 这样管理员不会在左侧菜单中看到“我的预约”

2. 工作台快捷入口按角色显示：
   - `student`、`teacher` 才显示“我的预约”
   - `admin` 不显示“我的预约”

3. 若已有基于 `to.meta.roles` 的前端守卫，则管理员手动输入 `/lab/my-bookings` 时会自动被拦截

#### 后端改造

修改文件：

- `D:\bishe\lamp_back\src\main\java\com\lamp\controller\LabController.java`

建议新增角色校验方法：

- `requireBookingUser()`

建议语义：

- 仅允许 `student`、`teacher` 使用个人预约相关接口
- `admin` 不允许进入个人预约流程

需收紧的接口：

| 接口 | 建议角色 |
| --- | --- |
| `POST /api/lab/booking` | `student`, `teacher` |
| `GET /api/lab/booking/my` | `student`, `teacher` |
| `PUT /api/lab/booking/{id}/cancel` | `student`, `teacher` |

这样可以保证管理员前后端都无法进入“我的预约”能力。

### 4.2 删除重复的实验室管理入口

#### 前端改造

修改文件：

- `D:\bishe\lamp_front\src\router\routes.js`
- `D:\bishe\lamp_front\src\views\admin\LabManageView.vue`

推荐方案：

1. 保留 `/lab/manage`
   - 原因：这个页面已经是真正的实验室管理页
   - 组件：`src/views/lab/LabManageView.vue`
   - 已有完整表格和管理逻辑

2. 删除 `/admin/lab` 路由
   - 因为它只是一个跳转壳页面，不承载真实业务

3. 删除或废弃 `src/views/admin/LabManageView.vue`
   - 若后续不再使用，可直接移除
   - 若暂时不想删文件，也可以先不再挂路由，让其不再出现在菜单中

#### 后端改造

后端无需改动。

原因：

- 两个“实验室管理”重复仅来源于前端路由与页面结构
- 实际后端管理接口只有一套 `/api/lab/manage/*`

### 4.3 管理员菜单的最终目标形态

调整完成后，管理员左侧菜单建议保留为：

- 工作台
- 考勤管理
- 实验室列表
- 预约审批
- 实验室管理
- 用户管理
- 个人资料

管理员不应再看到：

- 考勤签到
- 考勤记录
- 请假申请
- 我的请假
- 我的预约
- 重复的实验室管理

## 6. 推荐实施顺序

1. 先改前端路由：
   - 给 `/lab/my-bookings` 增加角色限制
   - 删除 `/admin/lab` 路由

2. 再改工作台入口：
   - 管理员不显示“我的预约”

3. 再改后端接口权限：
   - 收紧个人预约相关接口到 `student`、`teacher`

4. 最后清理冗余页面：
   - 移除或废弃 `src/views/admin/LabManageView.vue`

## 7. 验证方案

### 7.1 管理员账号验证

管理员登录后应满足：

- 看不到 `考勤签到`
- 看不到 `考勤记录`
- 看不到 `请假申请`
- 看不到 `我的请假`
- 看不到 `我的预约`
- 只看到一个 `实验室管理`

并且：

- 访问 `/lab/my-bookings` 会被前端拦截
- 即使绕过前端直接调用 `/api/lab/booking/my`，后端也返回无权限

### 7.2 学生 / 教师回归验证

学生或教师登录后应满足：

- 仍然可以正常进入“我的预约”
- 仍然可以创建预约和取消预约
- 实验室管理入口不会对非管理员开放

## 8. 结论

本次问题的本质不是业务功能缺失，而是“管理员个人功能没有完全收口”和“管理员实验室管理入口设计重复”。

按本方案调整后，可以达到以下效果：

- 管理员界面更干净，职责边界更清晰
- 前端菜单、路由、后端接口权限保持一致
- 消除重复实验室管理入口，减少用户混淆
- 不影响学生、教师原有预约流程
