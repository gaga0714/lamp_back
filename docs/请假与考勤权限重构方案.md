# 请假与考勤权限重构方案

## 1. 目标

本方案将系统中的考勤与请假模块按角色重新划分为以下职责：

- `student`：负责签到、签退、查看个人考勤、发起请假、查看自己的请假记录
- `teacher`：负责查看待审批请假并完成审批
- `admin`：仅负责查看考勤管理，不参与请假申请与请假审批

本次采用最小可实现方案，不新增数据库表，不增加师生绑定关系。当前语义为：任意教师都可以审批任意学生发起的请假申请。

## 2. 角色边界

### 2.1 学生

- 可访问：
  - `POST /api/attendance/check-in`
  - `GET /api/attendance/records`
  - `POST /api/attendance/leave`
  - `GET /api/attendance/leave/list`
- 前端菜单：
  - `考勤签到`
  - `考勤记录`
  - `请假申请`
  - `我的请假`

### 2.2 教师

- 可访问：
  - `GET /api/attendance/leave/pending`
  - `PUT /api/attendance/leave/{id}/approve`
- 前端菜单：
  - `请假审批`

### 2.3 管理员

- 可访问：
  - `GET /api/attendance/manage`
- 前端菜单：
  - `考勤管理`

## 3. 后端设计

### 3.1 角色校验策略

后端继续沿用 JWT + `UserContext` 的方式，不引入 Spring Security 的方法级注解，统一在控制器中手动校验角色。

在 `AttendanceController` 中增加以下内部方法：

- `requireLogin()`
- `requireRole(String... roles)`
- `requireStudent()`
- `requireTeacher()`
- `requireAdmin()`

这样可与现有 `LabController` 的权限风格保持一致，并避免出现“前端菜单限制了，但接口仍可直接调用”的问题。

### 3.2 接口调整

| 接口 | 角色 | 说明 |
| --- | --- | --- |
| `POST /api/attendance/check-in` | `student` | 学生签到/签退 |
| `GET /api/attendance/records` | `student` | 学生查看个人考勤 |
| `POST /api/attendance/leave` | `student` | 学生发起请假 |
| `GET /api/attendance/leave/list` | `student` | 学生查看自己的请假记录 |
| `GET /api/attendance/leave/pending` | `teacher` | 教师查看待审批请假列表 |
| `PUT /api/attendance/leave/{id}/approve` | `teacher` | 教师审批请假 |
| `GET /api/attendance/manage` | `admin` | 管理员查看考勤管理列表 |

### 3.3 教师待审批列表

教师待审批列表接口复用 `AttendanceService#getLeaveManageList()`，查询条件为：

- 仅返回 `status = 待审批` 的请假记录
- 支持分页

返回字段在原有请假记录基础上补充以下信息，方便教师审批页面直接展示：

- `username`
- `name`
- `approveRemark`
- `createTime`

### 3.4 审批业务规则

在 `AttendanceService#approveLeave()` 中增加业务校验：

1. 请假记录必须存在
2. 请假状态必须仍为 `待审批`
3. 被审批记录的申请人必须是 `student`

这样可以保证：

- 教师只能审批学生的请假
- 教师和管理员自己的请假记录不会进入审批流

### 3.5 数据库

本次方案不修改数据库结构，原因如下：

- 当前只需要按 `role` 做角色边界控制
- 当前不要求“导师只能审批自己学生”的复杂关系
- 现有 `leave_apply.user_id` 已足够支持最小可实现版本

如果后续要升级为“指定教师审批指定学生”，建议新增师生关联字段或中间表，再对请假查询与审批逻辑做精细化过滤。

## 4. 前端设计

### 4.1 路由权限

前端在 `src/router/routes.js` 中按角色收紧如下页面：

| 路由 | 角色 |
| --- | --- |
| `/attendance/check-in` | `student` |
| `/attendance/records` | `student` |
| `/attendance/leave/apply` | `student` |
| `/attendance/leave/list` | `student` |
| `/attendance/leave/approve` | `teacher` |
| `/attendance/manage` | `admin` |

其中新增教师页面：

- `/attendance/leave/approve`

### 4.2 路由守卫

在 `src/router/index.js` 中补充基于 `to.meta.roles` 的前端角色校验：

- 未登录时跳转登录页
- 已登录但角色不匹配时跳转首页

这样可以防止用户通过手动输入 URL 访问无权限页面。

### 4.3 教师请假审批页面

新增页面：

- `src/views/attendance/LeaveApproveView.vue`

页面功能包括：

- 查看待审批请假分页列表
- 展示学生学号、姓名、请假类型、起止时间、请假原因、状态、申请时间
- 支持“通过 / 驳回”操作
- 支持填写审批备注

### 4.4 前端接口封装

在 `src/api/attendance.js` 中新增：

- `getPendingLeaveList(params)`

对应后端接口：

- `GET /api/attendance/leave/pending`

原有 `approveLeave(id, data)` 保持复用。

### 4.5 工作台入口

在 `src/views/dashboard/DashboardView.vue` 中按角色展示不同快捷入口：

- `student`：考勤签到、考勤记录、请假申请、我的请假
- `teacher`：请假审批
- `admin`：考勤管理

这样可以避免教师和管理员在工作台看到不属于自身职责的请假入口。

## 6. 验证方案

### 6.1 学生账号验证

使用学生账号登录后，应满足：

- 能进入 `考勤签到`
- 能进入 `考勤记录`
- 能进入 `请假申请`
- 能进入 `我的请假`
- 看不到 `请假审批`
- 看不到 `考勤管理`

### 6.2 教师账号验证

使用教师账号登录后，应满足：

- 能进入 `请假审批`
- 能查看待审批请假列表
- 能对请假执行通过或驳回
- 无法进入学生侧请假申请页面
- 无法进入考勤管理页面

### 6.3 管理员账号验证

使用管理员账号登录后，应满足：

- 能进入 `考勤管理`
- 无法进入学生侧请假申请/我的请假页面
- 无法进入教师请假审批页面

### 6.4 接口权限验证

建议分别用不同角色账号直接调用接口确认：

- 学生调用 `/api/attendance/leave/pending` 应返回无权限
- 教师调用 `/api/attendance/manage` 应返回无权限
- 管理员调用 `/api/attendance/leave` 和 `/api/attendance/leave/{id}/approve` 应返回无权限

## 7. 后续可扩展项

如果后续需要进一步完善请假审批流程，可继续扩展：

1. 新增“教师只审批自己学生”的师生关系模型
2. 在请假表中增加 `approve_user_id`、`approve_time`
3. 支持教师按学生姓名、时间范围筛选待审批记录
4. 在工作台展示真实的待审批数量与请假统计
