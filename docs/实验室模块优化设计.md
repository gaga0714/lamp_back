# 实验室模块优化设计

## 1. 设计背景

当前实验室模块已经具备基础的实验室展示与预约能力，但仍存在以下问题：

1. 前端页面中实验室编号（`id`）未展示，用户无法快速区分同名实验室。
2. 实验室名称过多且不统一，不利于形成稳定、真实的科研平台风格。
3. 实验室表缺少“可用设备”信息，无法支持按设备筛选实验室。
4. 当前实验室查询仅支持名称关键词，尚不能满足预约场景下按日期、时段、容量、位置、设备等条件筛选的需求。
5. 当前预约流程是“先选实验室再选日期和时段”，不够符合真实预约使用逻辑。

本次优化目标是：

- 完善实验室表结构
- 明确实验室名称规范
- 增强实验室筛选能力
- 优化预约业务流程
- 保持实现复杂度适中，适合毕业设计项目落地

## 2. 优化目标

### 2.1 功能目标

学生或教师预约实验室时，可以按以下条件进行筛选：

- 实验室名称
- 预约日期
- 预约时段
- 容量（capacity）
- 仅看可预约
- 实验室位置（location）
- 实验室可用设备

查询后，前端展示符合条件的实验室列表，用户再选择实验室进行预约。

### 2.2 展示目标

实验室列表或详情页中展示以下信息：

- 实验室编号（id）
- 实验室名称
- 实验室位置
- 实验室描述
- 容量
- 状态（可预约 / 维护中）
- 可用设备

## 3. 数据库设计优化

### 3.1 当前实验室表字段

当前 `lab` 表主要字段为：

- `id`
- `name`
- `location`
- `description`
- `capacity`
- `status`

已能满足基础展示，但还不够支撑设备筛选。

### 3.2 新增字段建议

建议在 `lab` 表新增字段：

#### `equipment_info`

用于存储该实验室内可用设备信息。

建议类型：

```sql
equipment_info VARCHAR(255) DEFAULT NULL COMMENT '实验室可用设备，多个设备用逗号分隔'
```

示例值：

- `高性能工作站,投影仪,示波器`
- `GPU服务器,白板,会议显示屏`
- `机械臂平台,工业相机,PLC控制器`

设计理由：

考虑到当前项目规模较小、实验室数量有限（50 条左右），使用一个字段保存设备信息即可满足：

- 页面展示
- 简单按设备筛选
- 数据初始化方便

不必一开始就拆成单独的设备表，避免实现复杂度过高。

### 3.3 实验室编号是否单独建字段

本次设计中，不新增编号字段，直接使用 `id` 作为实验室编号展示。

原因：

1. `id` 本身唯一，天然可作为编号
2. 目前实验室数量不大，使用 `id` 足够清晰
3. 前端展示时可直接显示为“实验室编号：12”

如后续需要更规范的格式，可在前端显示为：

- `LAB-001`
- `LAB-023`

但数据库层仍可继续使用 `id`。

### 3.4 实验室名称规范

根据需求，实验室名称不再无限制扩展，而是固定为 10 类科研平台风格名称。不同实验室通过编号加位置区分。

建议固定为以下 10 个名称：

1. 智能感知实验室
2. 数据智能实验室
3. 嵌入式系统实验室
4. 网络空间安全实验室
5. 机器视觉实验室
6. 云计算与大数据实验室
7. 机器人控制实验室
8. 软件工程实验室
9. 虚拟仿真实验室
10. 智慧医疗实验室

命名策略说明：

- 名称用于表达实验室类型或方向
- 同名实验室允许存在多个
- 唯一区分依赖：
  - `id`
  - `location`

例如：

- `实验室编号：12，智能感知实验室，四教南-306`
- `实验室编号：27，智能感知实验室，十教-318`

这样既真实，也更利于前端筛选和后续假数据生成。

## 4. 查询筛选设计

### 4.1 查询条件列表

学生/教师预约实验室时，查询表单建议包含以下字段：

| 条件 | 类型 | 说明 |
|------|------|------|
| 实验室名称 | 下拉选择 | 固定 10 个实验室名称 |
| 预约日期 | 日期选择器 | 必填或建议优先选择 |
| 预约时段 | 下拉选择 | 固定 4 个时段 |
| 容量 | 数值输入 / 下拉范围 | 表示最少需要容纳多少人 |
| 仅看可预约 | 开关 / 复选框 | 是否仅显示当前可预约实验室 |
| 位置 | 下拉或关键词输入 | 如一教、四教南、十二教 |
| 可用设备 | 下拉 / 多选 | 如投影仪、GPU服务器、机械臂等 |

### 4.2 各条件的业务含义

#### 1. 实验室名称

用于按实验室类别筛选，建议使用下拉框，避免用户输入不规范名称。

#### 2. 预约日期

用于确定某一天的可预约情况。

#### 3. 预约时段

建议固定为当前系统已有 4 个时段：

- 08:00-10:00
- 10:00-12:00
- 14:00-16:00
- 16:00-18:00

#### 4. 容量

表示“至少可容纳多少人”。

查询逻辑建议为：

```text
lab.capacity >= 用户输入人数
```

#### 5. 仅看可预约

如果勾选，则需要同时满足：

1. `lab.status = available`
2. 当前日期 + 当前时段下，没有已存在的 `pending / approved` 预约记录

#### 6. 位置

支持按楼栋或完整位置筛选，例如：

- 一教
- 四教南
- 十二教
- 四教南-306

#### 7. 可用设备

根据 `equipment_info` 做匹配。

例如用户选择“投影仪”，则展示包含该设备的实验室。

### 4.3 核心查询逻辑

最核心的预约查询逻辑可以定义为：

查询某日期、某时段下，满足名称、位置、容量、设备等条件，且当前可预约的实验室列表。

业务判断顺序建议为：

1. 先筛选实验室基础条件：
   - 名称
   - 位置
   - 容量
   - 设备
   - 状态
2. 再结合预约表判断：
   - 指定日期
   - 指定时段
   - 是否已被预约（`pending / approved`）

## 5. 后端实现方案

### 5.1 实体类调整

`Lab` 实体建议增加字段：

```java
private String equipmentInfo;
```

数据库字段映射建议：

```java
@Column(length = 255)
private String equipmentInfo;
```

### 5.2 接口参数设计

建议扩展实验室查询接口参数。

当前接口：

```http
GET /lab/list
```

建议支持参数：

```text
name
date
slot
minCapacity
onlyAvailable
location
equipment
page
pageSize
```

示例：

```http
GET /lab/list?name=智能感知实验室&date=2026-03-10&slot=14:00-16:00&minCapacity=30&onlyAvailable=true&location=四教南&equipment=投影仪
```

### 5.3 推荐的接口返回字段

实验室列表接口建议返回：

- `id`
- `name`
- `location`
- `description`
- `capacity`
- `status`
- `equipmentInfo`
- `currentSlotAvailable`（可选，布尔值）

其中 `currentSlotAvailable` 建议后端在用户传入 `date + slot` 时返回，表示：

- `true`：该实验室在当前选择日期与时段下可预约
- `false`：该实验室在当前选择日期与时段下不可预约

这样前端展示更直观。

### 5.4 Repository / Service 实现建议

由于当前项目数据规模小（实验室数量约 50），后端实现可采用简单可维护优先策略。

#### 方案一：Service 层组合过滤（推荐）

流程：

1. 先按基础条件从 `lab` 表查实验室
2. 再在 Service 层根据 `date + slot + booking` 判断是否可预约
3. 返回最终结果

优点：

- 实现简单
- 适合毕设
- 易调试

#### 方案二：JPA 动态查询 / Specification

适合后期复杂化，但当前阶段不是必须。

### 5.5 与预约表联动逻辑

当 `onlyAvailable=true` 且用户传入 `date + slot` 时，需要排除掉以下实验室：

- `lab.status != available`
- 在 `lab_booking` 中已存在同一实验室、同一日期、同一时段，且状态为：
  - `pending`
  - `approved`

即：

```text
如果实验室维护中 -> 不可预约
如果该时段已被占用 -> 不可预约
否则 -> 可预约
```

## 6. 前端实现方案

### 6.1 实验室列表页优化

当前列表页仅支持名称关键词搜索，不足以支撑预约筛选。建议将实验室列表页改造成“实验室预约查询页”。

查询表单建议增加：

- 实验室名称（下拉）
- 预约日期（日期选择器）
- 预约时段（下拉）
- 容量（数值输入）
- 仅看可预约（开关）
- 位置（下拉或输入）
- 可用设备（下拉或多选）

### 6.2 列表展示内容优化

每张实验室卡片或列表项建议展示：

- 实验室编号：`id`
- 实验室名称：`name`
- 实验室位置：`location`
- 容量：`capacity`
- 状态：`status`
- 可用设备：`equipmentInfo`
- 简介：`description`

若传入了 `date + slot`，还可以展示：

- 当前时段是否可预约

例如：

- 当前时段可预约
- 当前时段已占用
- 实验室维护中

### 6.3 预约页优化建议

当前预约页是“先选实验室，再选日期和时段”。建议优化为：

#### 推荐流程

1. 先在实验室列表页完成筛选
2. 用户在筛选结果中点击某实验室
3. 带着实验室 `id` 进入预约页
4. 预约页保留日期、时段、用途填写

这样预约流程更符合实际使用逻辑。

## 7. SQL 调整建议

### 7.1 为实验室表新增设备字段

```sql
ALTER TABLE lab
ADD COLUMN equipment_info VARCHAR(255) DEFAULT NULL COMMENT '实验室可用设备，多个设备用逗号分隔' AFTER description;
```

### 7.2 查询字段建议

如果后续想按设备模糊匹配，可直接使用：

```sql
SELECT *
FROM lab
WHERE equipment_info LIKE '%投影仪%';
```

由于实验室数量不大，这种实现方式足够支撑毕业设计项目。

## 8. 推荐展示规则

### 8.1 编号展示

前端直接展示：

- `实验室编号：{{ item.id }}`

如想更美观，可格式化成：

- `LAB-001`
- `LAB-023`

### 8.2 名称展示

名称固定十个，不再无限扩展。

### 8.3 设备展示

展示规则建议：

- 若设备较少：直接显示全部
- 若设备较多：只显示前 2~3 个，剩余用 tooltip 查看

例如：

- `高性能工作站、投影仪、示波器`
- 鼠标悬停显示完整设备列表

## 9. 推荐设备词库

为了便于筛选，建议设备名称标准化，避免同义表达过多。推荐统一使用以下词汇：

- 高性能工作站
- GPU服务器
- 投影仪
- 会议显示屏
- 工业相机
- 示波器
- 机械臂平台
- PLC控制器
- FPGA开发板
- 白板
- 录播设备
- 传感器平台
- 虚拟现实设备
- 3D打印机
- 服务器集群

前端筛选时可直接提供这些设备选项。

## 10. 实施优先级建议

### 第一阶段（必须完成）

1. `lab` 表增加 `equipment_info`
2. 后端返回 `id` 和 `equipmentInfo`
3. 前端展示实验室编号与设备信息
4. 查询条件增加：
   - 名称
   - 日期
   - 时段
   - 容量
   - 仅看可预约
   - 位置
   - 设备

### 第二阶段（优化体验）

1. 返回 `currentSlotAvailable`
2. 预约页改为从筛选结果进入
3. 设备改成多选筛选
4. 位置改为楼栋下拉

## 11. 最终方案结论

本次实验室模块优化的核心思路是：

- `id` 直接作为实验室编号展示
- 实验室名称控制在固定 10 类
- 在 `lab` 表增加 `equipment_info` 支持设备展示与筛选
- 将实验室查询条件扩展为：
  - 实验室名称
  - 预约日期
  - 预约时段
  - 容量
  - 仅看可预约
  - 位置
  - 可用设备
- 最终实现“先筛选实验室，再预约实验室”的用户流程

该方案兼顾了：

- 业务合理性
- 毕业设计可实现性
- 前后端改造成本
- 页面展示效果

适合作为当前项目的实验室模块优化方向。
